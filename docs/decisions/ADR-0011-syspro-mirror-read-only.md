# ADR-0011 — Syspro mirror tables are read-only

**Status:** Accepted

**Settles:** open decision **D2**; addresses risk **R3**
([`../07-roadmap.md`](../07-roadmap.md)).

## Context

The project contains machine-exported **Syspro ERP mirror tables** (e.g.
`dbo.InvBuyer`): no foreign keys, a distinct machine style, and **re-generated by the ERP
export** — any hand or tool edit would be clobbered on the next export. The question (D2):
treat them as read-only, exclude them entirely, or allow editing?

`pr_*` extension tables can declare foreign keys that **reference** mirror tables, so the
mirror tables carry real diagram value as edge endpoints.

## Decision

Treat mirror tables as **read-only**:

- They **are** parsed and included in the model and the ERD, so their columns, primary key,
  and any edges to/from them render correctly.
- The tool **never edits and never rewrites** a read-only table's file — its bytes are left
  untouched (which also dovetails with the "never rewrite untouched files" rule of
  [ADR-0010](ADR-0010-formatting-strategy-lazy-canonicalization.md)).
- Read-only status is determined by **explicit configuration** (a path glob / list), **not**
  inferred from schema shape. Inference such as "has no FK ⇒ mirror" is rejected: it would
  wrongly freeze a brand-new empty `pr_*` table and is contrary to the project's
  no-heuristics ethos (cf. C10 / [ADR-0008](ADR-0008-fk-only-relationships.md)).

## Rationale

- **Excluding** them would turn FKs from `pr_*` tables into dangling edges and lose diagram
  context. **Editing** them is futile (clobbered by the next export). Read-only keeps full
  diagram fidelity with zero clobber risk.
- Explicit configuration is deterministic and auditable; it never silently mislabels a
  table.

## Consequences

- The model carries a per-table `readOnly` flag (default `false`). Phase 3 edit operations
  must refuse to modify a `readOnly` table, and the emitter/file-writer must never write one.
- In Phase 0 the flag is metadata only — parsing and round-trip behavior are identical; it
  affects rendering (Phase 1) and editing (Phase 3).
- The exact default classification rule (e.g. a `syspro/**` glob, or a configured list) is a
  follow-up to be pinned with project configuration; the policy — read-only via explicit
  config — is fixed here.

## Alternatives considered

- **Exclude entirely:** rejected — loses edges/endpoints and diagram context.
- **Fully editable:** rejected — edits are clobbered by the ERP re-export (R3).
- **Infer read-only from "no FKs":** rejected — misclassifies new/empty extension tables and
  relies on a heuristic.
